# DP-203 Skill Guide

## Data Quality Contracts
Data quality contracts define the schema, ranges, null expectations, uniqueness rules, freshness targets, and semantic meaning that downstream machine learning pipelines can rely on. DP-203 work matters to ML failures because model behavior often changes after a data pipeline silently changes. FailureLens IQ should use this topic when `data_quality_signals` mention null spikes, duplicate rows, schema drift, type conversion, stale partitions, or unexpected category values. A useful contract names both the technical rule and the business assumption behind it. For example, a timestamp field may be required because training labels must arrive after events, not before them. Strong contracts include automated checks, owner routing, severity levels, and exception handling. When quality evidence is sparse, the diagnosis should avoid claiming a root cause and should recommend human review. The enterprise lesson is that data reliability is part of model reliability. Teams cannot debug ML experiments well when upstream data meaning is undocumented. ## Feature Pipeline Lineage
Feature pipeline lineage records where each feature came from, which transformations produced it, and which version was used in training, validation, and deployment. DP-203 skills support this by encouraging clear data flow, versioned transformations, and repeatable orchestration. FailureLens IQ should inspect `preprocessing_steps`, `feature_set`, `training_config`, and `deployment_context` for evidence that a feature was generated differently across environments. Lineage is especially important when an experiment suspects leakage, because post-outcome fields, aggregate windows, and target-derived features can enter a pipeline accidentally. A feature can also become invalid when a source table changes grain or a join multiplies rows. Good remediation includes documenting feature definitions, adding tests for training-serving skew, and comparing feature distributions across stages. The diagnosis should say which assumption broke: the feature values used for evaluation matched values available at prediction time. Without lineage, teams are forced to remember incidents informally, which is exactly the organizational memory gap FailureLens IQ addresses. ## Batch Processing and Freshness
Batch processing pipelines often fail ML teams through stale data, delayed labels, partial loads, or inconsistent windows. DP-203 practitioners define service-level expectations for freshness and monitor whether each dataset meets them. In an experiment record, evidence may appear as `data_quality_signals` describing stale partitions, `error_logs` from failed jobs, or `deployment_context` showing a model serving on old features. Freshness is not only a data engineering metric; it changes model validity. A churn model using week-old usage features or a demand model missing recent events may degrade even if the training code is unchanged. FailureLens IQ should connect freshness evidence to metric degradation and drift indicators before diagnosing deployment drift. Remediation can include partition completeness checks, late-data handling, backfill procedures, and dashboards for feature availability. The best learning artifact explains how freshness affected the experiment's target and what check would have caught the issue before model evaluation or deployment. ## Join Keys and Grain Mismatch
Many ML failures begin with a join that looks technically valid but changes the dataset grain. A customer table joined to an events table can duplicate labels, overweight active users, or leak future behavior into training rows. DP-203 data modeling skills require teams to document primary keys, foreign keys, aggregation windows, and row-level meaning. FailureLens IQ should flag data quality and leakage risk when notes mention duplicates, exploding row counts, mismatched joins, or aggregate features that include outcome periods. The evidence should name the field or pipeline step involved, not just say "bad join." A strong diagnostic recommendation includes row count reconciliation, uniqueness checks, date boundary tests, and sampling joined records for semantic review. If the experiment has strong metrics but suspicious features, the system should consider data leakage. If the evidence is incomplete, confidence should drop. Teams improve when they learn to test grain assumptions systematically rather than rediscovering join problems during every failed experiment. ## Data Drift in Source Systems
Data drift can originate upstream when source systems change product instrumentation, logging behavior, data entry workflows, or customer mix. DP-203 knowledge helps teams trace whether a drift signal reflects real-world change or a pipeline artifact. FailureLens IQ should examine `drift_indicators`, `data_quality_signals`, and `deployment_context` together. A drift alert with no quality issue may indicate a legitimate population shift. A drift alert plus null spikes or schema changes may indicate broken data capture. A good diagnosis distinguishes these possibilities and records counter-evidence. Remediation includes source owner review, schema registry updates, distribution monitoring, and retraining criteria only after the data meaning is confirmed. Data drift should not automatically trigger retraining. Retraining on corrupted data can make the system worse. DP-203 practice asks teams to validate the data path first, then decide whether the model, features, labels, or monitoring thresholds require changes. This prevents expensive but shallow fixes. ## Label Generation Pipelines
Label generation pipelines define target outcomes, observation windows, exclusion rules, and timing constraints. Machine learning experiments fail when labels are noisy, late, inconsistent, or generated with information unavailable at prediction time. DP-203 skills are relevant because labels are data products with lineage and quality requirements. FailureLens IQ should use this topic when `target`, `validation_strategy`, `training_config`, or `engineer_notes` mention delayed outcomes, manual review labels, changing definitions, or suspiciously strong validation metrics. A precise diagnosis might say that the label window overlapped with feature extraction, creating temporal leakage, or that labels arrived too late for current monitoring to measure degradation. Remediation includes target definition documentation, label freshness checks, leakage boundary tests, and agreement metrics for human-labeled data. The system should not assume bad labels from poor performance alone. It should cite concrete signals and recommend focused checks. Reliable labels are foundational because every downstream metric inherits their assumptions. ## Reproducible Data Pipelines
Reproducible data pipelines let a team rebuild the training dataset from documented inputs, code versions, parameters, and time windows. DP-203 practices such as versioned storage, deterministic transformations, metadata capture, and environment separation help ML teams compare experiments fairly. FailureLens IQ should look for reproducibility gaps when `training_config` is incomplete, `preprocessing_steps` are vague, or metrics cannot be compared to baseline. A failed experiment without reproducible data teaches little because the team cannot know whether performance changed due to code, data, labels, features, or evaluation. Remediation should include dataset snapshots, pipeline run identifiers, schema validation, and traceable feature definitions. For enterprise trust, every diagnosis must be auditable. Reproducibility also supports responsible AI because subgroup evaluations need the same data version as the primary metrics. The manager dashboard can reveal teams that repeatedly fail from data process gaps, which points toward DP-203 learning rather than another modeling tutorial. ## Data Security and Access Boundaries
Data security influences ML experimentation when sensitive attributes, protected data, customer identifiers, or restricted sources are used for training and analysis. DP-203 guidance includes access control, least privilege, data masking, secure storage, and audit logging. FailureLens IQ should consider this knowledge when experiments mention protected attributes, privacy constraints, missing demographic labels, or manual exports. Security restrictions may explain why fairness analysis is incomplete, but they do not remove the need for responsible review. A good diagnostic response says what evidence is unavailable and why that limitation matters. Remediation might include approved aggregated fairness datasets, privacy-preserving analysis, role-based access, and documented exception workflows. Teams should not copy sensitive data into ad hoc notebooks to debug quickly. That creates a new failure. The system's safety language should keep recommendations at the team and process level, require approval for sensitive actions, and preserve an audit trail of what was considered.
