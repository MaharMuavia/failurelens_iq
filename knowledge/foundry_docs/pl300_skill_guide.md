# PL-300 Skill Guide

## Manager Dashboards and Metric Definitions
PL-300 practice teaches that dashboards must define metrics clearly enough for leaders to act without guessing. In FailureLens IQ, manager dashboards show failure rate, heatmaps, trend lines, vulnerability level, recurring pattern alerts, knowledge gaps, and recommended actions. Each metric should include a denominator, time window, and interpretation. Failure rate means failures divided by total experiments, not failures divided by only reviewed experiments. A failure heatmap counts categories after classification, so it should not include unknowns as if they were confirmed root causes. Trend points should use stable dates and categories. Manager-facing analytics must avoid individual blame and focus on team learning systems. The dashboard should answer whether the team is improving, stagnant, or worsening, while preserving evidence links for deeper review. PL-300 design also reminds builders that visual polish cannot substitute for semantic clarity. A beautiful chart with unclear definitions will create the same organizational forgetting that the product is meant to solve. ## Failure Heatmaps
Failure heatmaps help managers spot repeated categories across teams, projects, or time windows. In FailureLens IQ, categories such as Responsible AI, Data Leakage, Evaluation Methodology, Deployment Drift, Feature Engineering, and Data Quality become operational signals. PL-300 guidance encourages careful color use, consistent category labels, and sorting that supports comparison. A heatmap should not imply precision beyond the underlying data. If classification confidence is low, the dashboard should either exclude the item from confirmed counts or show uncertainty. Strong interpretation connects repeated patterns to skill gaps, process gates, or tooling needs. For example, three Responsible AI failures in thirty days suggest a review and training problem, not a one-off incident. The manager agent can convert the heatmap into a recommended action such as adding fairness review to sprint acceptance criteria. The visual should make the pattern easy to scan while preserving drill-down paths into the reasoning trace. ## Trend Lines and Learning Velocity
Trend lines show whether failure patterns are changing over time. PL-300 work requires choosing a stable grain, such as daily or weekly counts, and avoiding misleading visual scales. FailureLens IQ uses a thirty-day trend to estimate learning velocity. Improving means repeated failure categories are decreasing or remediations are followed by fewer similar incidents. Stagnant means the team sees the same issue without meaningful reduction. Worsening means frequency, severity, or vulnerability increases. A trend line should be interpreted with context because a team running more experiments may produce more failures while actually learning faster. The manager insight should compare failure count, total experiments, category repetition, and assessment evidence when available. A good dashboard invites action: schedule a learning intervention, revise a validation checklist, or route a responsible AI review. It should not shame a team. The purpose is to reveal where organizational memory is working and where it is still leaking. ## Certification Relevance Scores
Certification relevance scores help translate experiment failures into structured learning paths. PL-300 dashboards can show cert codes such as DP-100, AI-102, DP-203, and PL-300 with relevance between zero and one. The score should reflect failure category, knowledge retrieval citations, current certifications, recurrence, and manager priorities. FailureLens IQ should avoid recommending a certification merely because it appears in a document. The recommendation should connect to a concrete knowledge gap, such as imbalanced evaluation, fairness assessment, data quality contracts, or dashboard metric definitions. If a team member already holds a certification, the cert mapper should recommend a fallback or more specific practice path. Manager-facing visuals should show why a cert matters now. For example, AI-102 becomes urgent when responsible AI failures recur and grounding citations point to fairness controls. PL-300 itself is relevant when teams cannot see trends or when leaders lack trustworthy metrics for learning velocity and vulnerability. ## Executive Summary Cards
Executive summary cards should be concise, evidence-backed, and action-oriented. PL-300 guidance warns against filling dashboards with decorative cards that repeat the same number without context. In FailureLens IQ, a summary card may show vulnerability level, recurring pattern alert, recommended action, or human review volume. Each card should answer a specific management question. Is there a critical pattern? Which certification or process should be prioritized? Are low-confidence diagnoses accumulating? Does the team need immediate review before deployment? Good cards use restrained color semantics: red for critical risk, amber for warning, green for improvement, and muted colors for context. They should include a plain-language interpretation and a route to details. Summary cards are not substitutes for reasoning traces. They are entry points that help managers decide where to spend attention. The safest language remains team-level and process-level, because the product is designed for learning rather than blame. ## Drill-Through Analysis
Drill-through analysis lets a manager move from an aggregate chart to the specific experiments and reasoning evidence behind it. PL-300 reports become trustworthy when every number can be traced to rows and definitions. FailureLens IQ should allow a failure category count to lead to experiment IDs, classifier rules, diagnostic evidence, confidence factors, and knowledge citations. This matters because managers may challenge whether a recurring pattern is real. A drill-through path can show that three incidents share protected-attribute fairness gaps, or that several deployment drift failures share stale features. Drill-through also supports coaching because the manager can see the exact skill gap without reading every raw log. The design should preserve context during navigation: team, time window, category, and confidence. If a diagnosis was halted by the confidence gate, drill-through should show the halt reason rather than hiding the item. Honest uncertainty is more valuable than a false aggregate. ## Storytelling With Responsible AI Controls
PL-300 storytelling is not marketing language; it is the disciplined sequence of context, evidence, interpretation, and decision. For FailureLens IQ, a responsible story might begin with a team trend, show a recurring fairness pattern, drill into experiment evidence, explain the confidence level, and end with a human-approved action. The report should never imply automated production decisions. It should also avoid language that attributes failure to a person. Good storytelling separates observed data from inference. A chart can show that Responsible AI failures increased. The interpretation can say this suggests a process gap in fairness review. The recommendation can propose AI-102 learning and a sprint checklist update. Those are different levels of certainty. PL-300 skills help make that distinction visible through headings, annotations, legends, and report structure. The result is a dashboard that managers can trust during a hackathon demo and, more importantly, during real operational learning. ## Report Accessibility and Dense Layout
Enterprise dashboards need dense information without sacrificing accessibility. PL-300 practice includes readable typography, contrast, keyboard navigation, meaningful labels, and chart alternatives. FailureLens IQ's frontend should use compact tables and charts because ML managers compare many experiments, but density must not make the interface cryptic. Category colors should be accompanied by text labels. Badges should not be the only way to understand status. Tables need stable columns for experiment ID, team, stage, outcome, category, and timestamp. Charts should have tooltips or supporting lists that expose exact counts. Accessibility also supports responsible AI because affected stakeholders may need to review evidence. A dashboard that only works visually or hides key context in color fails the transparency goal. The best PL-300-style interface helps users scan, compare, drill down, and take action while preserving the audit trail and confidence limits behind every recommendation.
