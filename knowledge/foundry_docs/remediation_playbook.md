# Remediation Playbook

## Micro-Learning Plans
Micro-learning plans are short, focused interventions for teams that need a quick correction tied to a specific failure. A three-day plan should include one concept review, one evidence review using the failed experiment, and one practical change to the team's checklist or code review process. FailureLens IQ should choose a micro plan when confidence is adequate, the gap is narrow, and production risk is low or already contained. For an evaluation methodology failure, micro-learning might cover minority F1, precision-recall curves, and a validation checklist. For responsible AI, it may cover fairness metric definitions and model card updates, but human review remains mandatory. The plan should cite grounding documents and connect every task to the experiment fields that triggered the diagnosis. A micro plan is not a generic tutorial. It is an organizational memory artifact that helps the team avoid repeating the same failure in the next sprint. ## Balanced Seven-Day Plans
Balanced plans combine learning, implementation, and verification over a week. They are appropriate when a team has a repeated pattern, a moderate vulnerability level, or a failure that spans process and technical skills. FailureLens IQ's seven-day plan should include reading, a hands-on lab, peer review, and a manager-visible process update. For deployment drift, the plan might start with monitoring concepts, add a feature freshness lab, and end with an alert ownership checklist. For data quality, it might include quality contracts, schema tests, and source owner review. The plan should identify deliverables that can be inspected: updated validation report, new quality check, model card section, or dashboard annotation. Balanced plans work best when they avoid overload. The goal is behavior change, not certification cramming. Recommendations should remain team-level and should not imply that one engineer needs correction because a system failed. ## Immersive Remediation
Immersive remediation is used for critical vulnerability, repeated failures, or responsible AI risk that requires deeper change. It may span multiple sprints and include formal certification preparation, architecture review, process redesign, and manager follow-up. FailureLens IQ should recommend immersive work when the same category appears three or more times in thirty days, confidence is strong, and the business risk is high. For TEAM-B style recurring Responsible AI failures, the plan could include AI-102 study, fairness assessment labs, policy review, and a required model card gate before deployment. For repeated leakage, it could include feature store governance and temporal validation workshops. Immersive plans need explicit manager notes because time must be protected. They should also define success criteria, such as fewer repeated incidents, improved assessment scores, or completed checklist adoption. The plan should still be concrete and evidence-grounded, not a vague request to "improve ML maturity." ## Hands-On Labs
Hands-on labs turn a diagnosis into practice. A good lab uses the failed experiment's category, metrics, validation strategy, or feature set so the team rehearses the exact skill it needs. For imbalanced evaluation, the lab might ask participants to compare accuracy, minority F1, ROC AUC, and precision-recall AUC on a skewed dataset, then choose a threshold based on cost. For data leakage, the lab could remove a suspected feature and rerun validation with a time-based split. For responsible AI, the lab could inspect subgroup error rates and update a model card. FailureLens IQ should include one lab in every remediation result, with an experiment connection that explains why it matters. Labs should avoid using real private data in the demo product. Synthetic data is enough to practice reasoning. The output should be something managers can inspect, such as a short evidence table, updated checklist, or pull request template. ## Manager Notes
Manager notes translate technical findings into operational action. They should tell a manager what pattern was detected, why it matters, what support the team needs, and how progress should be measured. FailureLens IQ should keep manager notes concise and free of individual blame. A note for evaluation methodology might recommend adding minority-class metric review to sprint acceptance. A note for data quality might request ownership for upstream contracts. A note for responsible AI should require policy review and human approval before deployment. Manager notes should reference vulnerability level and recurring patterns when available. They should not overstate certainty. If the confidence gate halted, the manager note should ask for evidence collection rather than training. Good manager notes help leaders create the conditions for learning: time, checklist changes, peer review, and tooling. This is how failed experiments become permanent organizational memory instead of private debugging stories. ## Responsible AI Notices
Responsible AI notices are required when experiments involve fairness, bias, protected attributes, demographic gaps, disparate impact, or discrimination language. The notice should state that organizational policy and qualified human review are required before deployment or model updates. It should not claim legal compliance or make automated decisions. FailureLens IQ should include the notice in remediation results, reports, and frontend banners whenever `responsible_ai_flagged` is true. The notice should be paired with evidence, such as which field contained fairness language or which metric showed subgroup harm. Remediation may include model card updates, stakeholder review, fairness metric selection, sensitive attribute handling, and monitoring. If evidence is limited, the system should still route to review because fairness uncertainty is itself a risk. The tone should be careful and team-centered. Responsible AI remediation protects affected users and protects the team from trusting a model before governance evidence is complete. ## Confidence-Gated Remediation
Confidence-gated remediation means the system only produces strong learning and action plans when evidence supports the diagnosis. When confidence is below the dynamic threshold, FailureLens IQ should halt downstream recommendations or mark them as low confidence, require human review, and record the halt reason. This is especially important for sparse experiments, contradictory signals, or weak knowledge retrieval. A low-confidence case should not receive a confident root cause, certification mapping, or immersive plan. The correct remediation may be evidence collection: add metrics, baseline, validation strategy, logs, data quality signals, and deployment context. The report should say that human review is required and name the missing evidence. Confidence gating makes the product credible because it knows what it does not know. It also prevents teams from learning the wrong lesson. In enterprise ML, a cautious halt is often more valuable than a polished but unsupported recommendation. ## Sprint-Adjusted Learning
Sprint-adjusted learning fits remediation into the team's current delivery rhythm. A recommendation that ignores sprint commitments may be accurate but unused. FailureLens IQ should use team context, recurring patterns, vulnerability level, and failure severity to choose between small checklist changes, a one-week learning path, or a larger process intervention. Sprint-adjusted plans should identify immediate next actions, owner roles, and review points without assigning blame. For example, a team with repeated data leakage failures might add a temporal feature availability check to every experiment review this sprint, then schedule deeper feature lineage work later. A team with a critical Responsible AI pattern may need to pause deployment work until review is complete. The manager dashboard should connect these plans to learning velocity so leaders can see whether interventions reduce repeated failures. This closes the loop from experiment diagnosis to team improvement.
