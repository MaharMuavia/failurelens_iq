# ML Failure Taxonomy

## Data Leakage Failure
Data leakage occurs when training or validation uses information that would not be available at prediction time. Common examples include post-outcome fields, future aggregates, target-derived encodings, duplicated labels, and validation splits that mix future observations into past training. FailureLens IQ should classify leakage when `suspected_leakage_columns` is present, when `failure_observation` mentions leakage, or when metrics are unusually strong during evaluation but degrade after deployment. Evidence should cite exact fields, such as a feature name or validation strategy. Counter-evidence matters too: a suspicious feature may be legitimate if it is available before prediction and documented in lineage. Leakage diagnoses should avoid blame language because leakage is usually a pipeline control failure. Remediation includes feature audits, temporal boundary tests, training-serving availability checks, and reproducing metrics after removing risky columns. The root cause should be stated as an assumption failure: validation data represented production-time information when the feature set included unavailable information. ## Responsible AI or Bias Failure
Responsible AI or bias failures occur when model behavior may unfairly affect protected or sensitive groups, when fairness evidence is missing, or when governance controls were bypassed. Signals include words such as bias, fairness, protected, demographic, disparate, discrimination, equalized odds, demographic parity, and model card gaps. FailureLens IQ should flag responsible AI risk even when classification confidence is uncertain, because fairness review requires caution. The diagnosis should name process gaps such as missing subgroup metrics, unreviewed sensitive attributes, unclear approval, or absent transparency documentation. It should not accuse an individual. Evidence may come from `failure_observation`, `engineer_notes`, `feature_set`, `metrics`, or `deployment_context`. Remediation should include human-in-the-loop review, model card updates, policy-approved fairness metrics, and stakeholder sign-off before deployment. If evidence is sparse, the confidence gate should halt and request review. Responsible AI notices should be preserved in reports and SSE traces. ## Deployment and Drift Failure
Deployment and drift failures occur when a model that looked acceptable offline performs worse in production because data, labels, features, user behavior, or system integrations changed. Signals include drift indicators, monitoring alerts, degraded production metrics, stale features, schema changes, and new populations. FailureLens IQ should inspect `pipeline_stage`, `drift_indicators`, `deployment_context`, `data_quality_signals`, and `metric_degradation_score`. Drift is not a root cause by itself; it is evidence that the environment may no longer match training assumptions. The system should distinguish data drift, concept drift, training-serving skew, and pipeline breakage when possible. Remediation can include monitoring thresholds, retraining criteria, rollback plans, feature freshness checks, and ownership for drift triage. A strong diagnosis explains what changed and how it affected the target or features. If labels have not arrived, the system should state uncertainty and avoid claiming performance impact beyond available evidence. ## Evaluation Methodology Failure
Evaluation methodology failures happen when validation design, metric selection, thresholding, or baseline comparison does not match the deployment decision. Common examples include using accuracy on imbalanced data, random splits on time-ordered data, missing minority class metrics, comparing against an irrelevant baseline, or treating a single holdout as conclusive. FailureLens IQ should classify this category when `class_balance` is skewed, `minority_f1` is weak, `validation_strategy` is unsuitable, or `reported_metric` hides an important subgroup outcome. Evidence should cite metric values and validation choices. The violated assumption is often that the reported metric represented real operational quality. Remediation includes stratified validation when appropriate, precision-recall analysis, fold-level reporting, threshold review, and stronger holdout design. The diagnosis should not say the model "failed because accuracy is bad." It should explain that the evaluation process selected or trusted a metric that did not measure the failure mode that mattered. ## Feature Engineering Failure
Feature engineering failures occur when features are weak, unstable, mis-specified, unavailable at serving time, poorly transformed, or inconsistent with model assumptions. Signals include sparse feature sets, missing preprocessing, high-cardinality encodings, unstable feature importance, training-serving skew, and engineer notes about features not capturing the target. FailureLens IQ should inspect `preprocessing_steps`, `feature_set`, `training_config`, `data_quality_signals`, and model type. A feature problem differs from data leakage because the issue may be insufficient signal rather than forbidden information. It differs from data quality because raw data may be valid while transformations are not useful. Remediation can include feature definition review, ablation tests, SHAP or permutation importance, encoding changes, normalization, text preprocessing, or domain-informed features. The system should cite exact features or steps when possible. If there is no evidence about features, confidence should remain modest. Good learning captures which feature assumption failed and how to test it next. ## Data Quality Failure
Data quality failures arise when the data used by an experiment is incomplete, inconsistent, stale, duplicated, mislabeled, or semantically changed. Signals include null spikes, schema drift, duplicate rows, invalid ranges, class label noise, missing partitions, and source system errors. FailureLens IQ should classify data quality when `data_quality_signals` or `error_logs` provide direct evidence, or when deployment context shows stale or malformed inputs. A data quality diagnosis should distinguish raw data defects from feature transformation choices and from real population drift. Remediation includes quality contracts, schema checks, row count reconciliation, freshness monitoring, label audits, and source owner review. The root cause should cite field-level evidence and explain its implication for the model. If a metric dropped but quality signals are absent, the system should not invent a quality issue. It may recommend quality checks as next steps while keeping the primary diagnosis tied to stronger evidence. ## Unknown or Insufficient Evidence
Unknown is a valid category when the experiment lacks enough evidence to support a specific root cause. FailureLens IQ should use this category for sparse experiments, missing metrics, absent baseline, vague notes, or contradictory signals that cannot be resolved with local knowledge. Unknown is not a failure of the product; it is a safety feature. The confidence gate should require human review and prevent fabricated diagnoses. The root cause should use explicit language such as "Insufficient evidence to determine root cause with required confidence." The trace should record which agents were skipped, which fields were missing, and what evidence would be needed next. Remediation may include collecting logs, metrics, validation details, and data quality checks before training recommendations are issued. This protects teams from over-learning the wrong lesson. Organizational memory is only useful when it preserves uncertainty honestly instead of converting every incident into a confident story. ## Cross-Category Conflict Resolution
Real ML failures often contain signals from multiple categories. A biased model may also have data quality problems. A deployment drift incident may reveal weak evaluation methodology. A leakage issue may look like excellent feature engineering until production metrics fall. FailureLens IQ should evaluate all six classification rules and record conflicts rather than hiding them. Conflict resolution should prioritize direct evidence, safety-sensitive categories, metric degradation, and retrieval grounding. Responsible AI signals should always add review requirements even if another category is primary. Data leakage with explicit suspected columns may outrank weaker feature engineering evidence. Unknown should be selected when conflicts are strong and evidence is thin. The classification result should explain the resolution in plain language with field citations. This makes the agent more than a classifier: it plans, compares hypotheses, records counter-evidence, checks confidence, and routes uncertain cases to review instead of forcing a brittle label.
