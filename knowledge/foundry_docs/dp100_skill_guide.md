# DP-100 Skill Guide

## Imbalanced Classification Metrics
Accuracy is a weak headline metric when the positive or operationally important class is rare. A model can report ninety percent accuracy on a 90/10 dataset while missing every minority example, so DP-100 style evaluation should inspect minority class recall, minority F1, precision, false negative cost, confusion matrix shape, and threshold behavior. In FailureLens IQ, this topic is relevant when an experiment reports high accuracy but low `minority_f1`, an uneven `class_balance`, or engineer notes that mention missed high-risk cases. Strong practice is to choose the metric that matches the decision consequence before comparing models. For fraud, safety, churn, or medical triage, recall and precision-recall AUC often explain failure better than ROC AUC alone. Baseline metrics must be included because a drop from the previous production model may matter more than a single absolute score. Evidence should separate what the metric shows from why it happened. ## Stratified K-Fold Validation
Stratified k-fold validation preserves class proportions across folds, which makes it an important DP-100 technique for imbalanced classification. Without stratification, one validation fold may contain too few minority examples to estimate recall or F1 reliably, while another fold may contain an unusually easy sample. A misleading holdout can make an experiment appear stable even though the model never learned the underrepresented class. FailureLens IQ should treat validation strategy as evidence when `validation_strategy` is plain holdout, random split, or unspecified on a skewed `class_balance`. The recommended practice is to use stratification for independent and identically distributed classification data, record fold-level metrics, and inspect variance across folds. If fold-level minority F1 swings widely, the team should slow down before deployment. When data is time ordered, stratification may be inappropriate, and the safer approach is a time-aware split with explicit drift checks. ## ROC AUC, Accuracy, and Precision-Recall Tradeoffs
ROC AUC can look healthy on a skewed dataset because the false positive rate is normalized by a large negative class. Precision-recall curves often show the operational tradeoff more clearly because they focus on positive predictions and missed positives. Accuracy can hide both problems. A DP-100 candidate should understand that threshold selection is part of evaluation, not an afterthought. If `metrics` show strong AUC and weak minority F1, the failure may be methodology rather than model architecture. Teams should review calibration, threshold curves, business cost, and confusion matrices before claiming a model improved. The best experiment record includes the chosen operating point and why it was chosen. FailureLens IQ can ground recommendations in this distinction by citing the model's reported metric, the baseline metric, and the observed class balance. This helps the diagnostic agent avoid blaming people and instead identify a process gap in evaluation design. ## Model Drift Detection and Azure ML Monitoring
Deployment drift occurs when the relationship between training data and production data changes after release. Azure ML monitoring patterns emphasize comparing feature distributions, prediction distributions, target availability, and performance once labels arrive. Data drift alone is not always harmful, but drift combined with degraded recall, rising error rates, or new customer segments is a strong investigation signal. FailureLens IQ should examine `drift_indicators`, `deployment_context`, `pipeline_stage`, and `metric_degradation_score` when diagnosing monitoring failures. A practical DP-100 remediation plan includes baseline windows, alert thresholds, ownership of triage, and retraining criteria. Teams should log the exact features that drifted, the population affected, and whether the deployed model still satisfies reliability expectations. Monitoring also requires data quality checks because schema changes and null spikes can masquerade as concept drift. Good systems preserve enough context to recreate the deployment decision later. ## Deployment Readiness Checklists
Deployment readiness is a structured review that asks whether an experiment can safely become a production model. A checklist should include validation design, baseline comparison, responsible AI review, data quality status, feature lineage, reproducibility, rollback plan, model card updates, monitoring configuration, and owner assignment. In FailureLens IQ, missing deployment context or a deployment-stage failure with weak monitoring evidence should lower confidence and may trigger human review. DP-100 practice treats deployment as an engineering decision rather than a score celebration. A model with excellent offline accuracy can still be unready if its training data differs from production, if the minority class has poor recall, or if observability is absent. The checklist should connect each go/no-go item to evidence. That makes the recommendation auditable and prevents vague lessons such as "test more." The useful learning is precise: which readiness gate failed, what evidence showed it, and what must change next sprint. ## Experiment Tracking and Lineage
Experiment tracking records parameters, code version, dataset version, feature set, metrics, artifacts, and approval notes so teams can compare runs honestly. Lineage matters because many failures come from silent changes: a preprocessing step moved, a leakage column appeared, a baseline was calculated on different data, or a training configuration was copied from another task. DP-100 guidance emphasizes reproducibility and traceability across model development. FailureLens IQ should use `training_config`, `preprocessing_steps`, `feature_set`, `baseline_metrics`, and `engineer_notes` to reconstruct what changed. If evidence is missing, the system should say so and route to review instead of inventing a root cause. Strong experiment memory captures negative results with the same care as successful ones. Failed experiments are especially valuable because they reveal brittle assumptions. The team should be able to search prior failures and see which configuration, data slice, and evaluation choice produced the lesson. ## Feature Importance and SHAP Values
Feature importance and SHAP values help explain which inputs influenced a model, but they are not automatic proof of causality. DP-100 practitioners use them to detect suspicious reliance on proxy variables, leakage columns, unstable features, or features that changed meaning between training and deployment. In FailureLens IQ, feature analysis is relevant when `suspected_leakage_columns` is non-empty, when `feature_set` includes post-outcome fields, or when data quality signals point to inconsistent feature generation. SHAP summaries should be compared across training, validation, and production slices. A feature that dominates validation performance may be a legitimate predictor, a proxy for a protected attribute, or a leaked label. The diagnostic agent should present the evidence and counter-evidence separately. Responsible remediation includes removing or quarantining risky features, rerunning validation, and documenting why the remaining feature set is acceptable. The goal is a trustworthy feature pipeline, not merely a higher metric. ## Time-Series Validation and Holdout Design
Time-series validation protects teams from training on the future. Random splits can leak future patterns into validation when observations are ordered by customer lifecycle, calendar season, system version, or market condition. DP-100 candidates should know when to use rolling windows, backtesting, forward chaining, and final untouched holdout periods. FailureLens IQ should inspect `validation_strategy`, `dataset_name`, `target`, and `failure_observation` for language about temporal leakage, late-arriving labels, seasonality, or production degradation after a random validation success. Holdout design also needs enough minority examples and representative slices. A holdout that excludes new geographies, low-volume segments, or recent product behavior can overstate readiness. The correct response is not always a more complex model; often it is a validation redesign that matches deployment reality. The report should name the broken assumption clearly: validation data represented future production conditions when it did not.
