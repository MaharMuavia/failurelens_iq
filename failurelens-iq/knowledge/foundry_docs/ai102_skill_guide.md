# AI-102 Skill Guide

## Microsoft Responsible AI Principles
Microsoft Responsible AI practice centers on fairness, reliability and safety, privacy and security, inclusiveness, transparency, and accountability. AI-102 work requires turning those principles into design controls, not treating them as slogans after evaluation is complete. In FailureLens IQ, this knowledge is relevant when an experiment mentions bias, fairness, protected attributes, demographic performance gaps, disparate impact, or missing human review. The correct diagnosis should focus on system and process conditions: missing slice metrics, absent model cards, unclear approval ownership, or unreviewed sensitive attributes. Responsible AI evidence must be cited from fields such as `failure_observation`, `engineer_notes`, `metrics`, and `deployment_context`. A remediation plan should require human approval before production decisions. The system should never assign individual blame or imply that a single developer caused unfair behavior. The useful lesson is which governance control failed and how the team can make the next review more reliable. ## Bias Detection Methods
Bias detection begins by defining protected or sensitive attributes, affected groups, task context, and harm scenarios. Methods include group performance comparison, false positive and false negative parity checks, calibration by group, data representation review, counterfactual testing, and qualitative review with domain experts. AI-102 guidance emphasizes that a model can be globally accurate and still produce unacceptable outcomes for a subgroup. FailureLens IQ should treat references to demographic slices, protected terms, or fairness incidents as strong evidence for responsible AI review. If the experiment lacks group labels because privacy policy restricts collection, the diagnosis should explain the limitation rather than invent a fairness metric. Good practice is to document what was measured, what could not be measured, what proxy signals were considered, and who must approve next steps. Bias detection is not a one-time report. It is a recurring monitoring practice connected to model changes, data shifts, and user feedback. ## Demographic Parity and Equalized Odds
Demographic parity checks whether positive prediction rates are similar across groups, while equalized odds checks whether error rates are similar across groups conditional on the true label. Neither metric is universally correct. The right fairness target depends on the use case, legal context, harm model, and product policy. AI-102 learners should understand that optimizing one fairness metric can worsen another, and that fairness cannot be reduced to a single score. FailureLens IQ can use this section when an experiment mentions demographic parity, equal opportunity, equalized odds, protected class, or disparate outcomes. The diagnostic output should state the violated assumption precisely, such as "aggregate accuracy represented subgroup performance." Evidence should include group metrics when available and human review when unavailable. Remediation may include threshold review, reweighting, additional data collection, stakeholder review, or redesigning the decision workflow. A responsible recommendation always includes governance and approval, not only code changes. ## Counterfactual Fairness and Sensitive Attribute Handling
Counterfactual fairness asks whether a model's prediction would change if a protected attribute changed while relevant non-sensitive facts stayed the same. In practice, it is difficult because features are correlated with social conditions, geography, access patterns, and historical outcomes. AI-102 work therefore treats sensitive attribute handling as a design decision requiring privacy review, fairness review, and clear purpose limitation. FailureLens IQ should flag risk when `feature_set` or notes include protected attributes, proxies, demographic categories, or unexplained exclusion of fairness data. Removing a sensitive column does not automatically remove bias because proxies can remain. The diagnosis should separate evidence of direct use from evidence of proxy behavior. Remediation can include feature audits, counterfactual tests, documentation of allowed use, and a human-in-the-loop approval path. The safest language is process-focused: the pipeline lacked sufficient fairness evidence for the deployment decision, so review is required before action. ## Model Cards and Transparency Requirements
Model cards summarize intended use, training data, evaluation results, ethical considerations, limitations, monitoring plans, and approval status. AI-102 guidance treats transparency as a practical engineering artifact. Stakeholders need to know where a model works, where it does not, and which populations or scenarios require caution. FailureLens IQ should recommend model card updates when experiments expose hidden assumptions, missing subgroup metrics, weak validation, or deployment drift. A model card should include evidence tables rather than broad claims. For a failed experiment, the card can record the failure mode, root cause, mitigation, and remaining uncertainty. This converts debugging into organizational memory. Transparency also helps managers see repeated patterns, such as the same team encountering fairness review gaps across several sprints. The diagnostic agent should cite grounding sources and experiment fields, while the confidence gate should stop unsupported conclusions. Honest limitation language builds trust more than confident but weak claims. ## Human-in-the-Loop Design
Human-in-the-loop design assigns meaningful review points where human judgment can prevent harm, correct uncertainty, and approve sensitive decisions. AI-102 scenarios often require escalation when confidence is low, fairness signals are present, or production impact is high. FailureLens IQ should treat human review as a first-class outcome, not as an error. When `requires_human_review` is true, the report should name the reason, the evidence gap, and the role that should review it. Human review is not useful if reviewers receive only a raw JSON dump. They need a concise hypothesis, evidence, counter-evidence, confidence factors, and recommended next checks. For fairness incidents, reviewers should include product, policy, domain, and ML perspectives. The system must avoid automated production decisions. Learning recommendations can be generated, but deployment or remediation actions require approval. This approach protects teams from over-trusting local heuristic reasoning when the evidence is sparse or ethically sensitive. ## Fairness-Accuracy Tradeoffs
Fairness and accuracy can conflict, but the relationship is not always a simple tradeoff. Better data quality, clearer labels, subgroup-specific evaluation, and calibrated thresholds can improve both. AI-102 practice asks teams to identify the harm of each error type and decide which tradeoff is acceptable under policy. FailureLens IQ should use this topic when metrics show aggregate improvement but fairness notes report subgroup degradation. The diagnosis should not say that fairness is merely a cost. It should explain which assumption failed, such as "the selected metric represented all affected populations." Remediation may include collecting representative data, threshold tuning by policy-approved criteria, rejecting a feature set, or pausing deployment until stakeholders review impact. The manager dashboard should surface recurring fairness failures because repeated incidents suggest a capability gap, not an isolated experiment issue. ## Audit Trails for Responsible AI Decisions
Responsible AI decisions need audit trails that preserve evidence, reasoning, approvals, and unresolved risks. An audit trail should show which agent or reviewer made a recommendation, what inputs were considered, what output was produced, and how long the step took. FailureLens IQ's `AuditEntry` and `AgentTraceEntry` structures support this by recording actions, reasoning steps, evidence fields, confidence, grounding citations, and skip reasons. AI-102 guidance values this trace because fairness decisions may be questioned later by managers, customers, auditors, or affected users. The audit trail should distinguish fact from inference. It should also record when confidence was too low and the pipeline halted. This prevents quiet fabrication and demonstrates accountability. For responsible AI failures, every recommendation should include a notice that organizational policy and human approval are required. The goal is not to automate ethics; it is to make the review process visible, repeatable, and evidence-grounded.
